# 대한민국 부동산 경매/공매 추천 에이전트 — 설계 문서

## 1. 목표

사용자가 텔레그램으로 대화하듯 조건(지역, 예산, 물건 유형)을 말하면, 그 조건에 맞는
부동산 경매·공매 물건을 검색해서 추천해주는 에이전트. 여러 사용자가 각자 조건을
저장해두고, 조건에 맞는 신규 물건이 뜨면 알림을 받을 수도 있다.

이 저장소의 기존 패턴(`briefing.py`, `stock_picker.py`, `politics_briefing.py`)이
"매일 배치 실행 → 정적 HTML 생성 → 텔레그램 알림"이었다면, 이번 에이전트는
"상시 대기 → 사용자가 말 걸면 그때 검색해서 답변"하는 대화형 서비스라는 점이
가장 큰 차이다. 이 차이가 인프라 설계에 영향을 준다 (3.4절 참고).

## 2. 데이터 소스

### 2.1 온비드 공매 (1차 소스, 공식 API — 우선 구현)

한국자산관리공사(캠코)가 운영하는 온비드는 공공데이터포털(data.go.kr)에 공식
Open API를 제공한다. 회원가입 후 활용신청 → 승인 시 서비스키 발급이라는 절차이며
개발계정은 1일 약 1,000건 트래픽이 주어진다. 사용할 서비스:

- **차세대 온비드 부동산 물건목록 조회서비스** (`data.go.kr/data/15157207`) — 입찰
  진행/예정 중인 부동산 공매물건 목록. 유형(아파트/주택/상가/오피스텔/토지), 소재지,
  감정가, 최저입찰가 등 핵심 필드 제공.
- **차세대 온비드 물건상세 입찰정보 조회서비스** (`data.go.kr/data/15157251`) —
  회차별 입찰 방법, 경쟁 구조, 참가 제한 등 상세 정보.
- **온비드 코드 조회서비스** (`data.go.kr/data/15000920`) — 용도별 코드, 시/도/
  시군구 주소 코드. 검색 조건을 API 파라미터로 변환할 때 필요.

구현은 직접 REST 호출 대신 오픈소스 라이브러리
[`PublicDataReader`](https://github.com/WooilJeong/PublicDataReader)의 `Kamco`
모듈(`pdr.Kamco(serviceKey)`)을 사용한다. `stock_picker.py`가 같은 제작자의
`FinanceDataReader`를 쓰고 있어 기술 스택 일관성도 맞는다. 실제 신청 대상
서비스와 호출 함수는 `auction_agent/README.md` 1절 참고 (`물건정보` 서비스의
`통합용도별물건목록` 기능 — 아파트/주택/상가/토지 등 전 용도를 하나의 목록으로
반환한다).

주의: "공매"는 압류재산·국유재산 등을 다루며, 법원 "경매"와 법적으로 다른 절차다.
사용자가 원하는 것이 "경매"라면 공매만으로는 커버리지가 부족하므로 2.2가 필요하다.

### 2.2 법원경매정보 (2차 소스, 스크래핑 — 법적 검토 후 구현)

대법원 법원경매정보(courtauction.go.kr)는 공식 오픈API가 없다. 스크래핑이
사실상 유일한 방법인데:

- 이용약관과 `robots.txt`를 먼저 확인하고, 과도한 요청으로 서버에 부담을 주지
  않도록 요청 간격을 두어야 한다.
- 사이트 구조가 예고 없이 바뀌면 즉시 깨질 수 있어 유지보수 비용이 크다.
- 상업적 재배포나 대량 크롤링은 별도 법적 리스크가 있을 수 있다.

따라서 이 소스는 **기본값 비활성화**(`ENABLE_COURT_SCRAPING=false`)로 두고,
온비드만으로 충분한지 먼저 검증한 뒤 필요 시 켜는 것을 권장한다. 법적으로 더
안전한 대안은 지지옥션·굿옥션·탱크옥션 등 민간 유료 API/제휴로 전환하는 것이다
(권리분석까지 정제된 데이터를 제공하지만 월 구독료가 든다).

### 2.3 데이터 소스 비교

| | 온비드 공매 | 법원경매 스크래핑 | 민간 유료 API |
|---|---|---|---|
| 합법성/안정성 | 높음 (공식 API) | 중간 (ToS 검토 필요) | 높음 (계약 기반) |
| 커버리지 | 압류재산·국유재산 | 일반 부동산 경매 전체 | 전체 + 권리분석 |
| 비용 | 무료 | 무료 (개발 비용만) | 유료 (월 구독) |
| 권리분석 | 없음 | 없음 (직접 등기부 확인 필요) | 포함되는 경우 많음 |

## 3. 시스템 아키텍처

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────┐
│ 텔레그램 봇  │◄───►│  검색/추천 엔진   │◄───►│  데이터 소스 계층 │
│ (대화 UI)   │     │  (scorer.py)      │     │ onbid_source.py  │
└─────────────┘     └──────────────────┘     │ court_auction_.. │
       │                     │                └─────────────────┘
       ▼                     ▼
┌─────────────┐     ┌──────────────────┐
│ 사용자 프로필 │     │  알림 스케줄러    │
│ profiles.py  │     │ (APScheduler)     │
└─────────────┘     └──────────────────┘
```

### 3.1 공통 데이터 모델 (`models.py`)

온비드/법원경매 두 소스를 같은 형태로 다루기 위한 공통 스키마:
`source, item_id, title, property_type, region_sido, region_sigungu, address,
appraisal_price(감정가), min_bid_price(최저입찰가), bid_price_rate(최저가율),
failed_count(유찰횟수), bid_start_date, bid_end_date, area, status, source_url,
rights_note`.

### 3.2 추천/필터링 엔진 (`scorer.py`)

1단계는 규칙 기반 필터/스코어링으로 시작한다 (LLM 호출은 비용·지연이 있으므로
1차 필터링에는 쓰지 않는다):

- 예산 적합도: `min_bid_price <= budget_max`
- 지역 일치: `region_sido/sigungu`가 관심 지역에 포함되는지
- 유형 일치: 아파트/주택 vs 상가/오피스텔/토지
- 가격 메리트: `bid_price_rate`(낮을수록 유리), `failed_count`(유찰 횟수)
- 마감 임박도: `bid_end_date` 기준 정렬

상위 N개만 골라 OpenAI API(이미 `OPENAI_API_KEY`로 연동되어 있음)로 한 줄
추천 코멘트를 생성하는 것은 2단계 확장으로 남겨둔다.

### 3.3 권리분석 범위 (중요한 기대치 설정)

등기부등본 자동 조회(선순위 임차인, 근저당 등)는 대법원 인터넷등기소의 유료
발급 체계와 얽혀 있어 완전 자동화가 어렵다. MVP에서는:

- 유찰 횟수, 감정가 대비 낙찰가율 같은 "가격 리스크"만 자동 스코어링하고,
- "법적 리스크(권리분석)는 직접 등기부등본을 확인하세요"라는 안내 문구와
  법원경매정보/등기소 링크를 답변에 항상 포함한다.

이걸 넘어서는 자동 권리분석은 별도 유료 데이터 연동이 필요한 후속 과제로 분리한다.

### 3.4 인프라: 왜 GitHub Actions cron만으로는 부족한가

기존 `briefing.py` 계열은 GitHub Actions cron이 하루 한두 번 실행되고 끝나는
배치 작업이라 잘 맞는다. 하지만 "대화하면 바로 답하는" 텔레그램 봇은 폴링이든
웹훅이든 **상시 실행되는 프로세스**가 필요하다. GitHub Actions는 이런 장시간
상주 프로세스에는 적합하지 않다 (job 최대 실행시간 제한, 실행당 과금/쿼터).

권장 방향:

- **웹훅 방식** + 서버리스 함수(예: 오라클/AWS/Fly.io 무료 티어, Cloudflare
  Workers 등)에 배포 — 상시 서버 없이도 이벤트 기반으로 동작.
- 또는 소형 상시 인스턴스(예: 가정용 라즈베리파이, 저가 VPS)에서 폴링 방식 봇
  프로세스를 그냥 띄워두는 방법 (가장 단순하지만 어딘가에는 "항상 켜진 곳"이 필요).
- 알림(조건에 맞는 신규 물건 발생 시 푸시)만큼은 기존처럼 GitHub Actions cron이
  하루 1회 배치로 온비드를 조회해 각 사용자 프로필과 대조 후 텔레그램으로 보내는
  방식으로 유지할 수 있다. 즉 "대화형 즉시 검색"과 "정기 알림"은 실행 방식이
  달라 인프라를 분리하는 것이 현실적이다.

이번 스캐폴드는 어느 배포 방식을 택하든 재사용 가능하도록 봇 로직과 실행
엔트리포인트를 분리해뒀다 (`telegram_bot.py`는 polling으로 로컬 실행 가능,
동일 핸들러를 웹훅 서버에도 그대로 붙일 수 있음).

## 4. 다중 사용자 프로필

여러 사용자가 각자 조건(예산/관심지역/관심유형/알림주기)을 저장한다.
`profiles.py`는 텔레그램 `chat_id`를 키로 JSON에 저장한다.

**개인정보 주의**: 이 저장소는 `index.html`, `sitemap.xml` 등을 보면 알 수 있듯이
공개 웹사이트로 퍼블리시되는 저장소다. 사용자 예산·관심지역 같은 개인 데이터가
담긴 프로필 파일을 이 공개 레포에 커밋하면 안 된다. 따라서:

- `auction_agent/data/` 전체를 `.gitignore`에 추가했다.
- 실제 운영 시에는 프로필 저장 경로를 로컬 디스크가 아니라 별도 비공개
  스토리지(작은 DB, 비공개 Gist, KV 스토어 등)로 옮기는 것을 권장한다.
  `AUCTION_PROFILES_PATH` 환경변수로 경로를 바꿀 수 있게 해뒀다.

## 5. 텔레그램 명령어 설계

| 명령어 | 설명 |
|---|---|
| `/start` | 소개 및 사용법 안내 |
| `/검색 <유형> <지역> <예산>` | 즉시 검색 (예: `/검색 아파트 서울 강남 5억`) |
| `/조건저장` | 대화형으로 예산/지역/유형/알림주기를 물어보고 프로필 저장 |
| `/내조건` | 저장된 내 조건 확인 |
| `/조건삭제` | 저장된 조건 삭제 |
| `/알림on`, `/알림off` | 정기 알림 켜기/끄기 |

자연어 자유 입력("서울 강남 아파트 3억 이하, 유찰 1회 이상")도 파싱해서 위 검색
로직으로 연결하는 것은 2단계 확장(간단한 정규식 우선, 필요하면 OpenAI 함수
호출로 구조화)으로 남겨둔다.

## 6. 단계별 로드맵

1. **Phase 1 (MVP)**: 온비드 API 연동, 규칙 기반 스코어링, `/검색` 명령어만
   지원하는 단일 대화(프로필 저장 없이 매번 파라미터 입력).
2. **Phase 2**: 다중 사용자 프로필 저장, `/조건저장`~`/알림` 명령어, 일 1회
   배치 알림 (GitHub Actions cron 재사용).
3. **Phase 3**: 법원경매정보 스크래핑 추가 (법적 검토 통과 시) 로 커버리지 확대.
4. **Phase 4**: LLM 기반 추천 코멘트/자연어 조건 파싱, 유료 권리분석 API 연동 검토.

## 7. 파일 구조

```
auction_agent/
  DESIGN.md          # 본 설계 문서
  README.md          # 설정/실행 가이드 (온비드 키 신청, 봇 생성 방법)
  config.py          # 환경변수 로딩
  models.py          # 공통 데이터 모델 (AuctionItem, UserProfile)
  onbid_source.py    # 온비드 API 연동 (PublicDataReader.Kamco 래핑)
  court_auction_source.py  # 법원경매 스크래핑 스텁 (기본 비활성화)
  scorer.py          # 규칙 기반 필터링/스코어링
  profiles.py         # 사용자 프로필 저장/조회 (JSON, gitignore 처리)
  telegram_bot.py    # 텔레그램 봇 명령어 핸들러
  requirements.txt   # 추가 의존성
  data/               # 런타임 데이터 (git에 커밋되지 않음)
```
